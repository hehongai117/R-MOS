## 第一部分：项目现状

### 1.1 这是什么

R-MOS（Robot Maintenance Operating System）是面向职业院校/培训机构的**机器人维保实训平台**：教师上传机器人资料 → AI 分析生成教学内容（SOP、3D 模型、故障库）→ 学生在数字孪生环境中标准化实训 → 系统自动裁决评分 + 五维技能画像。当前处于 **MVP 阶段，机器人交互全部由 Mock 适配器模拟**。

### 1.5 部署基建现状（本次修订新增，已与仓库核对）

前一版文档称"无容器化/CI 部署"，**与实况不符**。仓库里已有：

- `r-mos-backend/Dockerfile`（uvicorn 双 worker）
- 根目录 `docker-compose.yml`（backend + postgres:14-alpine，含健康检查、环境变量注入）
- `.github/workflows/` 下 3 条 CI：`backend-ci.yml`、`frontend-ci.yml`、`integration-ci.yml`
- backend-ci / integration-ci 均起 **Postgres 16 服务**并先执行 `alembic upgrade head`

**真正缺的是**：CD（自动部署）、staging/prod 环境分离、生产校验接入部署流程；以及若干**版本漂移**（Dockerfile 用 python:3.11 而本地/CI 均为 3.13；compose 用 Postgres 14 而 CI 用 16）。此外 backend-ci 里 pytest 步骤把 `DATABASE_URL` 指向 SQLite（`backend-ci.yml` 第 60 行附近），导致仅限 Postgres 的门禁测试（如 `tests/unit/test_audit_query_index_gate.py`）在 CI 中被 skip——**Postgres 服务就在旁边跑着却没用上**，修复成本很低。

## 第三部分：已知问题与技术债（已与代码核对）

### 3.1 存储抽象泄漏（换对象存储前必须先修，被前一版低估）

`FileStorageBase` 接口本身在，但**本地文件系统语义已泄漏到下游**，"新增一个 S3 实现类"是不够的：

- `get_full_path()` 返回**本地磁盘路径**，且被重度依赖：`robots.py` 资产下发直接 `FileResponse(full_path)`；`cad_converter.py` / `pdf_extractor.py` / `manifest_generator.py` 的分析管线都要求文件真实在本地磁盘上。
- 有两处**完全绕过抽象层**的硬编码：`app/api/v1/endpoints/robots.py:471` 与 `app/services/training/workbench_draft_generator.py:72` 直接拼 `Path("data/robot-assets")`。
- 整个接口是**同步**的；换成 S3 网络 IO 后会阻塞 async 事件循环。

| 类别 | 问题 | 严重度 | 归属 |

### 战略岔路口（需业务先定方向，尽早对齐）

**S-1 真实机器人接入 vs 纯数字孪生**
- 适配器架构（`BaseRobotAdapter`）已为接真机预留。若客户有真机联动需求 → 实现 Real/Gazebo 适配器（最大一块工程）；若卖点是纯数字孪生 → 把孪生体验做深，不碰真机。**此决定影响半年研发重心。**

**S-2 多租户隔离（若走多校 SaaS）**
- 现状比想象更弱：`school_name` 只是 `user` 表上可空字符串，非外键。规模化前需硬化租户隔离（行级安全或按租户分 schema），确保校际数据零串。
- **零成本预防（现在就执行，不必等业务定方向）**：从本文发布起，所有新建表一律带租户维度字段；新查询不写跨租户逻辑。拖得越晚，数据模型迁移越贵。
