## 第一部分：项目现状

### 1.1 这是什么

R-MOS（Robot Maintenance Operating System）是面向职业院校/培训机构的**机器人维保实训平台**：教师上传机器人资料 → AI 分析生成教学内容（SOP、3D 模型、故障库）→ 学生在数字孪生环境中标准化实训 → 系统自动裁决评分 + 五维技能画像。当前处于 **MVP 阶段，机器人交互全部由 Mock 适配器模拟**。

### 1.2 当前健康度（已实测，2026-07-02）

| 项 | 状态 |
|----|------|
| 后端测试 | **692 passed / 3 skipped**（含 29 个 e2e；裸 `pytest` 已可干净运行） |
| 前端测试 | **465 passed / 2 skipped**（60 个测试文件） |
| 类型/词法 | `tsc --noEmit` + `eslint --max-warnings 0` 全绿 |
| 覆盖率门禁 | 后端三大端点 + 前端核心组件均纳入 CI 门禁并达标 |
| 生产依赖安全 | **0 漏洞**（axios 已升 1.18；dev 构建链尚有 12 个，需破坏性升级，择机处理） |
| 端到端实跑 | 已用真实 Postgres + 浏览器完整走通教师/学生黄金路径，**无崩溃** |

**运行环境已就绪**：本机 PostgreSQL 有 `rmos` 库（66 表、迁移到最新、含种子数据：11 用户 / 20 SOP / 11 机器人 / 2869 学校）。

### 1.3 近期已完成的工作

**质量硬化 4 阶段（均已合并入 `quality-hardening-phase2` 分支）：**
1. **仓库工程化与卫生** — 根目录清理、遗留分支清理、gitignore 加固。
2. **测试安全网** — 为 6 个待重构巨型文件建特征测试 + CI 覆盖率门禁。
3. **巨型文件重构** — 7 个巨型文件行为等价拆分（如 `SOPMaintenancePage` 1615→700、`agent.py` 1214→335），后端按业务域拆 sub-router，前端抽 hooks/子组件。
4. **性能与健壮性硬化** — 补路由级/3D 错误边界、API 瞬时失败重试、AI 调用超时；**性能测量工具已就绪但基线待采集**（见 `docs/superpowers/plans/phase4-baseline-collection-cheatsheet.md`）。

**测试规划暴露并修复的问题（详见 `docs/2026-07-02-full-test-plan.md`）：**
- 8 个 P0 真实功能 bug（此前被特征测试"锁定为错误行为"，全绿但行为错）——已全部修复。
- P1 测试基建（`pytest.ini` 未配 testpaths）、P2 依赖安全（axios）——已修复。
- 演示优化：默认机器人设为资产完整的 ATOM-01。

### 1.4 分支说明

- **`quality-hardening-phase2`**：当前主工作分支，**上述所有阶段与修复都在这里**（历史沿用链式合并：phase1→phase2→phase3→phase4→bugfix 均并入此分支）。

### 2.2 分层架构

```
前端 (React/Vite) ──HTTP REST + WebSocket──> 后端 (FastAPI)
                                              ├─ services/  业务服务(50+)
                                              ├─ adapters/  机器人适配器(Mock/预留物理)
                                              ├─ llm/       多提供商 LLM 路由
                                              └─ models/    SQLAlchemy ORM(32+)
                                                    │
                                    PostgreSQL + 本地文件存储(data/robot-assets/)
```

### 2.3 关键模块导航（接手先看这些）

**后端**
- `app/api/v1/endpoints/` — 37 个路由模块，共约 154 个 API。Phase 3 已按域拆分（如 `agent.py` + `agent_knowledge/evidence/v2/governance.py`）。
- `app/services/` — 业务逻辑核心。`training/`（训练）、`memory/`（技能画像）、`llm/`（模型路由）、`knowledge/`（RAG）、`orchestration/`（Agent 编排）。
- `app/adapters/` — `BaseRobotAdapter`（抽象）→ `MockRobotAdapter`。**接真机的扩展点。**
- `app/services/storage/` — `FileStorageBase`（抽象）→ `LocalFileStorage`。**接对象存储的扩展点。**
- `app/services/authz_guard.py` — RBAC 核心（`ActorContext`）。
- `main.py` — 应用入口，含全局异常处理器、可选计时中间件（`PERF_TIMING=1`）。

**前端**
- `src/pages/` — 页面。`SOPMaintenancePage`（实操工作台，已重构）、教师监控、学生技能画像等。
- `src/components/Viewer3D/` — 通用 3D 查看器（清单驱动）。`Atom01Interactive`（已重构）、`ManifestDrivenRenderer`。
- `src/adjudication/` — **SOP 裁决引擎（核心 IP）**：执行器 + 决策引擎 + 评分。
- `src/config/` — 配置驱动：`nav.ts`（导航）、`routes.ts`（路由权限）、`robots.ts`（含默认机器人）。
- `src/store/` — Zustand 状态（`authStore`、`robotContextStore`）。
- `src/api/` — Axios 客户端，`client.ts`（含 401 刷新）+ `retry.ts`（幂等重试）。

### 2.4 如何运行

```bash
# 后端（需 PostgreSQL 在跑，.env 里 DATABASE_URL 已配）
cd r-mos-backend && source venv/bin/activate && python main.py   # → :8000

# 前端（dev server 已配代理转发 /api 到 8000，无需额外配 .env）
cd r-mos-frontend && npm run dev                                  # → :3000
```

## 第三部分：已知问题与技术债

| 类别 | 问题 | 严重度 | 建议 |
|------|------|--------|------|
| 存储 | 3D/证据文件存本地磁盘，不支持多实例/扩容 | 高（阻碍规模化） | 见路线图梯队一 |
| 部署 | 无容器化/CI 部署，只有开发环境 | 高 | 见梯队一 |
| 可观测性 | 无集中日志/错误告警/性能监控 | 高 | 见梯队一 |
| 数据质量 | 部分机器人（如灵犀X1 id=3）3D 资产不全，发布态却缺文件 | 中 | 见梯队二（资产校验闸门） |
| 测试 | 特征测试曾"锁定错误行为"；无浏览器级 E2E；e2e 用 SQLite 与生产 Postgres 有方言差异 | 中 | 见梯队二 |
| 测试门禁 | `test_audit_query_index_gate` 等因无 CI Postgres 被 skip | 中 | 见梯队二 |
| 性能 | 从未量化（Phase 4 基线待采集） | 中 | 见梯队二 |
| 依赖 | dev 构建工具链 12 个漏洞（需 `npm audit fix --force` 破坏性升级） | 低 | 择机 |
| 技术债 | Pydantic V2 class-based Config 弃用告警；WS 遥测初连有一次抖动后自愈；Viewer3DErrorBoundary 切换机器人不自动恢复（需手动重试） | 低 | 顺手清理 |

## 第四部分：结构性升级路线图（三梯队）

> 排序原则：先"生产化"→再"质量闸门"→定"战略方向"→最后"规模化"。

### 梯队一 · 生产化地基（接第一个真实客户前必须）

**T1-1 对象存储替换本地磁盘**
- 现状：`LocalFileStorage`（`data/robot-assets/`）。扩展点已备好（`FileStorageBase`）。
- 目标：新增 `S3FileStorage`（S3/阿里云 OSS/MinIO 兼容），配置切换。资产 URL、上传、下载路径统一走抽象层。
- 收益：多实例部署、备份、扩容的前提。投入相对小，优先做。

**T1-2 容器化 + 环境分离 + CI/CD**
- Docker 化前后端 + docker-compose（含 Postgres）；分 dev/staging/prod；GitHub Actions 自动构建部署。
- 补齐生产校验（`SECRET_KEY` 非默认、强制 Postgres——代码已有校验逻辑，需接入部署流程）。

**T1-3 可观测性**
- 错误追踪（Sentry 前后端）、指标（Prometheus/OpenTelemetry）、把已有的计时中间件（`PERF_TIMING`）接成真实 APM、结构化日志集中化。

### 梯队二 · 交付质量体系（决定敢不敢承诺给学校用）

**T2-1 机器人资产完整性校验闸门**
- 机器人置为 `READY/发布` 前，自动校验 manifest + 必需 GLB 存在；缺失则阻断并明确报错。直接根治"课上到一半 3D 打不开"。

**T2-2 测试体系升级**
- 区分"特征测试(锁现状)"与"规格测试(断言正确)"，建立 bug 回归集。
- 引入 **Playwright 浏览器级 E2E**，覆盖登录→实操→提交→报告黄金路径。
- CI 用容器化 Postgres 跑（消除 SQLite 方言差异 + 解锁被 skip 的 DB 门禁）。

**T2-3 性能基线 → 针对性优化**
- 用 Phase 4 已备好的工具采基线（`phase4-baseline-collection-cheatsheet.md`），依数据定位真实瓶颈再优化，不臆测。

### 战略岔路口（需业务先定方向）

**S-1 真实机器人接入 vs 纯数字孪生**
- 适配器架构（`BaseRobotAdapter`）已为接真机预留。若客户有真机联动需求 → 实现 Real/Gazebo 适配器（最大一块工程）；若卖点是纯数字孪生 → 把孪生体验做深，不碰真机。**此决定影响半年研发重心。**

**S-2 多租户隔离（若走多校 SaaS）**
- 现按 `school_name` 区分。规模化前需硬化租户隔离（行级安全或按租户分 schema），确保校际数据零串。

### 梯队三 · 锦上添花（规模稳定后）

- AI 生成内容质量评测机制（保证不同机型 SOP 质量稳定）。
- 国际化（现界面文字写死中文）。
- 前端设计系统/组件库统一沉淀。
- `adjudication` 裁决引擎作为核心 IP 强化（加测试、文档化、可对外）。
- 清理 Pydantic V2 弃用、dev 依赖漏洞等遗留债。

## 第五部分：给后续开发的执行建议

1. **先合并主线**：决定把 `quality-hardening-phase2` 作为新主线，避免继续在语义混乱的分支上开发。
2. **沿用既有工作法**：本项目采用 spec → plan → subagent 逐 Task 执行、任务间 review 的方式（见 `docs/superpowers/`）。每个梯队项建议先写实施计划再落地。
3. **守住测试网**：Phase 2/3 建立的特征测试 + CI 门禁是重构安全网，改动务必保持全绿；新功能补规格测试。
4. **按梯队推进**：不要跳过梯队一直接做梯队三。生产化地基不牢，后面所有优化都不稳。
5. **战略先行**：S-1/S-2 两个岔路口尽早与业务对齐，它们决定后续架构走向。
